# Nepogano — поточна архітектура (технічний стан)

Цей документ фіксує, що вже реалізовано, як сутності зв'язані між собою і
які правила навмисно закладені — щоб не переплутати щось при наступних
змінах. `PRODUCT_BRIEF.md` — про філософію й напрямок продукту, цей файл —
про те, як воно влаштовано технічно **зараз**. Оновлювати разом зі змінами.

## Сутності бази даних і як вони зв'язані

| Таблиця | Що це | Власник/доступ |
|---|---|---|
| `checkins` | Особистий щоденний чек-ін (mood, note, photo, update_count) | тільки `user_id` (сам юзер) |
| `subjects` | "Сутність" — дитина/улюбленець/інше, окремий щоденник власника | тільки `owner_id`, за замовчуванням повністю приватно |
| `subject_checkins` | Чек-іни сутності — той самий формат, що `checkins` | через `subjects.owner_id`; read-only розширення нижче |
| `profiles` | Публічний профіль: `friend_code` (для додавання в друзі), `display_name` | власник + друзі (RLS) |
| `friendships` | **Симетрична дружба** — одна пара = один рядок (`requester_id`/`addressee_id`), видимість парна | обидві сторони пари |
| `friend_folders` | **"Коло" в UI** — персональне групування вже доданих друзів | тільки власник, БЕЗ спільного членства |
| `friend_folder_members` | Хто з моїх друзів у якому моєму колі | власник керує; учасник бачить лише факт свого членства |
| `subject_folder_shares` | Який щоденник сутності відкритий якому колу (лише перегляд) | керує власник сутності+кола |
| `circle_guesses` | Гра "вгадай настрій" — тільки між реальними людьми | guesser бачить свої здогадки; ціль бачить здогадки про себе |

## П'ять правил, які легко переплутати

1. **"Коло" ≠ спільний контейнер.** У UI слово "коло" використовується для
   `friend_folders` — але це суто персональний ярлик власника (як тег), без
   права запрошення й без спільного членства. Стара модель `circles`/
   `circle_members` (звідси й назва) була замінена саме тому, що спільні
   контейнери давали транзитивний витік видимості до незнайомців.

2. **Видимість чек-інів визначає лише `friendships`.** Не приналежність до
   одного кола. Дружба симетрична: якщо A і B — друзі, вони бачать чек-іни
   одне одного незалежно від того, в яких колах хто кого розкладає.

3. **Сутності приватні за замовчуванням.** Доступ до конкретної сутності
   можна відкрити конкретному колу — і це **тільки перегляд**
   (`subjects_select_shared`/`subject_checkins_select_shared`). Гра-вгадування
   (`circle_guesses`) **не** підключена до сутностей — вона працює лише між
   `auth.users`. Не змінювати це без окремого продуктового рішення.

4. **Старі `circles`/`circle_members` досі в БД, але код їх не використовує.**
   Свідомо не видалені під час переходу на `friendships` (закрите тестування
   вже йшло на них) — приберуться окремою міграцією, коли всі тестувальники
   гарантовано перейдуть на новий білд. Не плутати зі старим `join_circle_by_code`
   RPC — він теж мертвий, замінений на `add_friend_by_code`.

5. **Особисті дані з `auth.users` не читаються напряму клієнтом.** Email
   інших людей зберігається дублюючись прямо в рядках (`friendships.requester_email`/
   `addressee_email`), бо `auth.users` недоступна через PostgREST. Якщо
   знадобиться ще якесь поле "про іншу людину" — той самий трюк, або
   `profiles` + RLS-політика видимості (як `profiles_select_friends`).

## SQL-міграції (виконувати саме в цьому порядку)

Усі лежать у `docs/`, усі безпечні для повторного запуску (`if not exists`/
`drop ... if exists` скрізь, де можна було).

1. `friends-migration.sql` — профілі, дружба, кола, RLS-фікс видимості чек-інів, display name
2. `subjects-migration.sql` — сутності (залежить від `friends-migration.sql`: використовує ту саму delete_own_account)
3. `checkin-update-count-migration.sql` — лічильник "Оновлено N разів" (залежить від #2 — чіпає `subject_checkins`)
4. `guess-stats-migration.sql` — RLS для статистики вгадувань
5. `subject-sharing-migration.sql` — доступ колам до щоденників сутностей (залежить від #1 і #2)

## Огляд фіч по екранах

**Головний екран (`CheckInScreen`, `lib/main.dart`)** — чек-ін настрою (Ніяк/Непогано/Збс) + нотатка + фото, редагування, картка дня, тижнева стрічка. Перемикач "Я | [сутності] | +" зверху — та сама логіка екрану, просто інша `_table`/`_idColumn`/`_idValue` (checkins vs subject_checkins). Довгий тап на сутність → меню "Поділитись з колом" / "Видалити щоденник".

**Друзі (`FriendsScreen`, `lib/friends_screen.dart`)** — список друзів (парна дружба), кола-чіпи для сортування (без права запрошення), запрошення по коду/лінку/email, статистика вгадувань, секція "Спільні щоденники" (сутності, відкриті мені чужими власниками).

**Людина (`PersonDetailScreen`)** — до 7 останніх днів друга, вгадування настрою по кожному дню окремо.

**Історія (`HistoryScreen`)** — календар + ретроспектива місяця, теж мультисуб'єктна (своя або сутності, включно з чужою відкритою через `subject_folder_shares` — повністю read-only).

**Картка дня (`DayCardScreen`)** — шер зображення, показує лічильник оновлень, якщо є.

**Лендінг (`nepogano.app`, тека `landing/`)** — Vercel, `join.html` обробляє `/join/<code>` (App Links + Play Install Referrer для deferred deep link, якщо застосунку ще не було встановлено).

## Свідомо НЕ реалізовано (не пропонувати як "забутий баг")

- Історія для сутностей інших людей — тільки перегляд, без можливості лишити коментар чи вгадати.
- Створення кола прямо з вікна "Поділитись щоденником" — веде на екран Друзі.
- Multi-owner кола (будь-хто запрошує) — свідомо відкинуто, звідси й перехід на дружбу.
- QR-код для додавання друга — обговорювалось, відкладено.
